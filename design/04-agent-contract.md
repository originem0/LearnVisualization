# 04. Agent 契约 — 课程生成的规则

> 这份文件定义 Agent 生成课程时的硬约束。它直接喂给 prompt，也是 quality.py 的校验依据。

---

## 一、Agent 的角色

Agent 不是写手，不是百科搬运工。它是**课程设计助手**：

- 负责结构化输出，不负责最终教学判断
- 产出必须通过引擎校验 + 质量检查 + 人工审核
- 生成的内容是草稿，不是成品

---

## 二、Plan 阶段约束

Agent 生成课程计划时必须：

### 2.1 知识类型声明
- 每个模块必须标注 `knowledgeTypes`（至少一个）
- 整门课的模块不能全是同一知识类型
- 概念性和程序性必须交替出现（不能连续 5 个概念模块）

### 2.2 认知层级推进
- 课程的前 1/3 模块允许停留在 remember/understand
- 中间 1/3 必须达到 apply/analyze
- 最后 1/3 应有 evaluate/create 层级的模块
- 质量检查拒绝全课程只有 remember 层的设计

### 2.3 模块类型多样性
- 不允许连续 3 个相同 moduleKind
- system-overview 最多占总模块数的 30%
- 至少有一个非 mechanism-walkthrough 类型

### 2.4 焦点问题质量
- 必须是真正的问题（包含 ? 或 ？）
- 不能是"X 是什么"句式（定义式不是焦点问题）
- 必须指向认知冲突、权衡或机制选择
- 好："为什么 Git 选择快照而不是增量？"
- 差："Git 的对象模型是什么？"

---

## 三、Compose 阶段约束

Agent 生成单个模块时必须：

### 3.1 误解质量
- 必须描述一个**具体的错误直觉**，不是"很多人不理解 X"
- 好："认为 Git 存的是文件差异，像 diff 工具一样"
- 差："很多人对 Git 的版本管理缺乏深入理解"

### 3.2 逻辑链质量
- 每步必须因果依赖前一步（不是并列罗列）
- 总步数 4-8 步（太少说不清，太多记不住）
- 质量检查：如果删掉任何一步，链条应该断裂

### 3.3 叙事块要求
- 至少 6 个叙事块
- 必须包含至少 1 个 steps 块（worked example）
- 必须包含至少 1 个 callout 块
- 概念性模块建议包含 comparison 块
- 程序性模块建议包含 trace 或 annotated-example 块
- 元认知模块必须包含 reflection 块

### 3.4 例子质量
- 至少 2 个具体例子
- 例子必须包含领域实体（不能写"举个例子"或"比如说..."）
- 每个例子至少 20 个字符
- 不允许包含省略号占位符（"..." 在例子中是警告）

### 3.5 检索练习要求
- 至少 1 个检索练习
- 必须标注 bloomLevel
- 不允许全部是 remember 层
- answerShape 必须引导结构化思考（不是一个词的答案）
- 生成式优于选择式

### 3.6 桥接质量
- bridgeTo 必须指向一个**尚未解决的问题**
- 不能是"下一章讲 X"（预告式）
- 必须从当前章的结论自然延伸

### 3.7 练习系统（v3 新增）
- 程序性模块至少有 2 个练习，脚手架级别不同
- 概念性模块至少有 1 个分类或对比练习
- 练习的 bloomLevel 不能全低于模块的 bloomLevel

## 四、质量控制策略

> **架构决策（2026-03-22）**：`run_quality_checks` 已移除。内容质量由 prompt 层控制，不由校验层阻断。
>
> 原因：质量检查（模板化语言检测、examples 长度、steps 块要求等）在实际运行中反复误杀合法内容（哲学课程无 steps、人文主题 examples 短于 20 字），且检查结果无消费者——用户看不到 warning，也无法手动修复。唯一的效果是阻断导出。
>
> 当前策略：
> - **prompt 层**：在系统提示中注入学习科学理论（Merrill、Bloom、Sweller、Bjork、Mayer、Novak），让 LLM 理解 WHY 而非只遵守 WHAT，对任意主题都能做出正确的教学设计判断
> - **normalize 层**：结构完整性检查（必需字段存在性），缺失字段给 fallback 而非 raise
> - **engine 层**：仅 title 缺失和 narrative 为空是 error，其余降级为 warning
> - **前端层**：未知叙事块类型通过 fallback 渲染链映射到已有视觉模式，不丢失内容

## 五、生成负面样本

Agent prompt 应包含负面样本（告诉 Agent "不要这样做"）：

### 差的焦点问题
```
✗ "什么是 Transformer？" — 定义式，不触发冲突
✗ "深入理解注意力机制" — 不是问题
✓ "为什么 Attention 不用 CNN 的局部窗口，而要看全部位置？" — 指向设计决策
```

### 差的误解
```
✗ "很多人对注意力机制理解不深" — 空泛
✗ "注意力是一个重要概念" — 不是误解
✓ "认为注意力就是给每个词打分，分高的就重要" — 具体的错误心智模型
```

### 差的检索练习
```
✗ type: fill-gap, prompt: "Transformer 的全称是______" — 事实记忆
✗ type: fill-gap, prompt: "注意力机制的输入是______" — 术语回忆
✓ type: compare-variants, prompt: "如果把 Self-Attention 的 softmax 换成 hardmax（只保留最大值），对长句子的翻译会发生什么变化？为什么？" — 反事实推理
```

## 六、Human Review Gate

所有 Agent 生成的内容必须经过人工审核。审核不是检查格式（那是质量检查的事），而是判断：

1. 焦点问题是否真的尖锐？还是只是符合格式的平庸问题？
2. 误解是否真的是学习者常犯的？还是 Agent 编造的？
3. Worked example 是否真的帮助理解？还是只是走过场？
4. 概念图是否真的外显了关键结构？还是节点堆砌？
5. 课程有没有退化成"好看的文章"？

审核结果记录在 `review/approval.json`：
```json
{
  "approved": true,
  "reviewedBy": "reviewer-name",
  "reviewedAt": "ISO-8601",
  "notes": "审核意见"
}
```
