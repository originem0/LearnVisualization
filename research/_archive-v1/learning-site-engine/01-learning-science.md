# 01 — Learning Science / 认知机制

## 研究目标

这部分研究要回答的不是“教育学上什么都重要”，而是：

> **面对高复杂度、高元素交互性的知识，什么样的表达结构更有利于理解、记忆与迁移？**

这直接决定 Learning Site Engine 的内容骨架和交互规则。

---

## 核心判断

### 1. 复杂知识的难点不只是信息量大，而是“元素交互性高”
这类主题（比如 LLM、数据库内核、因果推断、操作系统）的问题不在于术语多，而在于概念之间相互依赖。

学习者不是只记单点，而是要同时追踪：

- 结构关系
- 因果方向
- 执行顺序
- 局部与整体映射

这正是认知负荷理论最有价值的地方：它提醒我们，**工作记忆不是无限容器**。如果把高交互性的内容直接堆成连续文字，学习者会很快“看懂字、看不懂系统”。

### 2. 对复杂内容，worked examples 和 step-by-step 推导非常关键
研究与教学实践反复说明：对新手而言，先给完整 worked example，通常比让他直接自己解决问题更有效。

对应到产品上，这意味着：

- 不应该一上来先给定义
- 要先给“问题 → 推导 → 结果”的过程
- 复杂过程应该切成离散步骤，每一步只改变一个关键变量

这和当前站点里的 `steps` block 高度一致，而且说明这条路不是审美偏好，而是认知上有依据。

### 3. 可视化不是装饰，它的价值在于“把关系外化”
复杂知识难，是因为关系结构需要在脑中临时维持。图示、概念图、对比图的意义，是把这些关系外部化，从而减少工作记忆负担。

但不是所有“图”都有用。多媒体学习相关原则反复强调：

- **coherence**：去掉无关装饰
- **signaling**：关键关系要被显式标注
- **segmenting**：连续过程要分段
- **contiguity**：相关信息要空间上靠近
- **redundancy**：不要把相同信息用低效方式重复堆叠

这意味着一个学习站的图示设计，不是“做漂亮图”，而是“让结构可见、减少无效负荷”。

### 4. retrieval practice 重要，但它不是第一阶段的全部重点
retrieval practice（主动回忆 / testing effect）对长期保留是有效的，特别是相对于单纯重读。

但课堂和真实教学情境下，研究也提醒我们：它虽然总体有益，却不总是无脑优于所有替代活动。

这对产品设计的启示是：

- 不能只有阅读
- 也不能简单加 quiz 就以为自己科学了
- retrieval 应该嵌在理解结构之后，而不是取代理解结构本身

所以对当前项目而言，优先级应该是：

1. 先把理解链设计好
2. 再在关键模块间加入 retrieval / self-check

### 5. 概念图非常适合高结构化知识，但它必须服务“意义建构”
概念图的价值不只是罗列节点，而是帮助学习者：

- 看到概念之间的关系
- 看到层次结构与依赖顺序
- 把新知识挂接到已有 schema 上

如果图只是“术语点点相连”，那只是视觉噪音。真正有用的概念图要回答：

- 哪些是核心节点？
- 哪些是 prerequisite？
- 哪些是因果、组成、对比、流程关系？

## 对产品的直接启示

### 应该固化为产品硬规则的内容

#### Rule 1 — 每个模块必须围绕一个焦点问题
不要定义驱动，要问题驱动。

#### Rule 2 — 每个模块至少有一个 worked example 或分步推导
尤其是高元素交互的内容，必须有推导过程。

#### Rule 3 — 每个模块先给关系全貌，再给局部展开
这就是概念图 / 结构图的存在理由。

#### Rule 4 — 图和文必须分工，不要互相复读
文字负责推导与解释；图负责关系显化；步骤卡负责过程切片。

#### Rule 5 — 轻交互应该服务验证理解，而不是增加噪声
交互不是玩具。它存在的前提是：能帮助用户操纵变量、控制节奏、立即看到反馈。

#### Rule 6 — retrieval 要作为第二层能力引入
不能一开始就把测试当主菜。先让用户形成 mental model，再让他回忆、迁移、检验。

#### Rule 7 — 概念图必须表达关系类型，而不是只表达“有关联”
引擎未来需要支持的不只是 node-edge，而是 prerequisite / contrast / flow / causal / part-of 等更细关系。

## 对当前 LearnVisualization 的判断

### 做对的地方
- 焦点问题驱动
- opening + narrative + bridge 的模块节奏是对的
- `steps` 叙事块方向很对
- concept map schema 化方向对
- 强调“先全貌再深入”是对的

### 还不够的地方
- retrieval / self-check 基本还没进入体系
- 交互组件虽然多，但认知目标还没被显式建模
- 概念图关系类型还不够细
- 还没形成“新手/中阶/专家”分层的节奏控制

---

## 暂定结论

Learning Site Engine 的教学底盘，至少应该建立在这些原则上：

1. **问题驱动**而不是定义驱动
2. **worked examples 优先**于抽象讲义
3. **关系外化**优先于文字堆砌
4. **分段与标示**优先于连续密集信息流
5. **交互验证**优先于装饰性交互
6. **retrieval 作为二阶段能力**引入
7. **概念图服务于 schema 建构**，不是视觉点缀
