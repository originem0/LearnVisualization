# 06. Design Philosophy Synthesis

> 目的：把当前已经沉淀下来的 research、中文参考材料、学习科学、引擎架构思考、Agent 架构思考，统一收束成一套 **可执行的设计思想**。  
> 这份文件不是最终的 `DESIGN.md v2`，而是它的上位母本。

---

## 一、先说结论：这个产品到底在做什么

这个产品不是：

- 知识集合站
- 好看一点的文章站
- 课程平台壳子
- 一键自动建站工具
- LLM 术语解释器

这个产品真正要做的是：

> **把复杂知识转化成“可被理解、可被穿透、可被迁移”的学习体验。**

更准确一点：

> **它是一个复杂知识的可视化学习站引擎。**

而它的输出，不应该只是页面，而应该是：

- 认知换挡
- 结构化理解
- 更高质量的问题意识
- 可迁移的知识模型

换句话说，未来不管是做 LLM、因果推断、PostgreSQL、操作系统还是别的主题，
都不应该只是“生成课程页”，而是生成：

**带来认知升级的学习结构。**

---

## 二、四条被吸收进来的主线

这轮输入材料虽然来源不同，但最后其实都收敛到了 4 条主线。

## 主线 1：启发不是信息增量，而是认知换挡

来自《认知升级大三元》的最关键贡献是：

- 启发感来自“解释力强且反常识”的知识
- 真正有价值的知识不是更多知识，而是“换挡知识”
- 问题、知识、认知不能被孤立理解，而要三位一体地看

### 被吸收成的设计思想

- 每章都必须有一个要击穿的旧成见
- 每章都必须有一个“换挡点”
- 每章不只是解释概念，而是重写用户的解释框架

所以这个产品的目标不是：

- 让用户知道更多名词

而是：

- 让用户进入新的认知挡位

---

## 主线 2：深度理解的核心不是更多文本，而是更好的认知负荷管理

来自《理解：深度理解，穿透复杂》的最关键贡献是：

- 理解不是记忆，它多出来的是“意义”
- 工作记忆是复杂理解的瓶颈
- 组块化是高手与新手的核心差别
- 认知负荷优化不是技巧，而是教学结构本身
- 概念地图是认知结构的外显工具
- AI 最适合做“理解的助产士”

### 被吸收成的设计思想

- 页面必须主动承担认知负荷调度
- 必须把高元素交互性的内容拆成更可处理的步骤
- 每章都应该帮助用户形成新的“稳定组块”
- 概念图不是插图，而是理解入口
- 交互不是炫技，而是帮助工作记忆分段处理复杂性

因此，这个产品不该只是：

- 展示复杂

而应该：

- 分解复杂
- 呈现复杂
- 穿透复杂

---

## 主线 3：网站必须被抽象成 Engine，而不是继续在专题站层面打转

来自 Learning Site Engine research 的最关键贡献是：

- 课程内容必须是结构化对象，不是单篇文档
- Engine 与 Course Package 必须分离
- 课程不是目录树，而应该逐步走向依赖图
- visual layer 和 interaction layer 需要 schema 化
- validation-first publishing 是必须保留的护栏

### 被吸收成的设计思想

- 页面只是引擎的一种渲染结果，不是产品本体
- 未来系统要围绕 `course -> module -> narrative blocks -> visuals -> interactions` 来建模
- 主题实例可以变化，但教学结构必须能复用
- 设计规范最终必须能落到 schema 与 pipeline 上，而不只是写在文档里

---

## 主线 4：Agent 不能是一键自动写站器，而必须是结构化课程生产系统

来自 Agent architecture research 的最关键贡献是：

- workflow 要先于 autonomous agent
- planner / composer / critic / review gate 必须分工
- 结构化输出比花哨文案重要
- human-in-the-loop 不能取消
- 最终要的是高质量课程产品，不是一次写成的 prose

### 被吸收成的设计思想

- Agent 的职责不是“写页面”，而是“组织课程生产”
- 未来生成链路必须分 stage
- 每一 stage 的输出要能被校验、被 review、被 revision
- 真正的自动化目标是流程自动化，而不是责任自动化

---

## 三、统一后的产品定义

综合上面四条主线，这个产品未来的定义可以写成：

> **Learning Site Engine 是一个面向复杂知识的可视化学习系统，它通过焦点问题、概念关系外显、分步推导、交互验证和模块桥接，把高元素交互性的知识转化为可理解、可迁移、可复审的学习体验。它调用 AI 作为结构化生产与认知放大器，但不把效率本身当作产品目标，而是服务于认知升级与创造性表达。**

它的生成入口是 Agent，但 Agent 只负责结构化生产，不替代教学审查。

---

## 四、统一后的核心设计原则

下面这些，才是应该进入未来 `DESIGN.md v2` 的“硬思想”。

## 1. 焦点问题优先于知识点罗列

每章必须只围绕一个焦点问题。

原因：
- 焦点问题提供认知聚焦
- 减少无关元素竞争工作记忆
- 有助于用户建立“这一章到底在解决什么”的问题图式

### 实际含义
不是“这一章讲 token、词表、BPE、特殊符号”，
而是“为什么模型不能直接读文本？”

---

## 2. 反直觉开场优先于定义开场

原因：
- 深度理解往往始于旧成见被打穿
- “换挡知识”要求先制造框架冲突

### 实际含义
优先打掉错误直觉，再给新框架。

---

## 3. 先外显结构，再进入局部推导

原因：
- 概念关系图能把本来要在脑中组织的关系提前外显
- 这有助于管理高元素交互性带来的负荷

### 实际含义
页面优先顺序应该是：

- 焦点问题
- 关键洞察
- 关系结构 / 路线
- 再进入正文和步骤

---

## 4. 每章必须有 worked example / steps

原因：
- worked example 是复杂理解的最低保障
- 用户不是靠定义学会的，而是靠看完整推演学会的

### 实际含义
每章都要有：
- steps
- 过程图
- worked example

而不是只有 prose。

---

## 5. 视觉元素必须承担认知任务

原因：
- 图、卡片、动画、交互如果不减负，就是噪声
- 多媒体学习不是摆一起，而是协同承担理解任务

### 实际含义
所有视觉元素必须能回答：
- 它减少了什么认知负荷？
- 它外显了什么关系？
- 它比文本多承担了什么？

---

## 6. 交互的价值在于“控制节奏 + 立即反馈 + 主动验证”

原因：
- 交互不是为了玩，而是为了让用户自己控制复杂过程的展开
- 这正是认知负荷理论中的 segmenting 与 worked-example 延伸

### 实际含义
交互组件必须优先支持：
- step-through
- compare
- simulate
- predict
- retrieve

---

## 7. 每章不仅要讲内容，还要帮助形成组块

原因：
- 深度理解最终依赖长时记忆中的高质量组块
- 教学不是传信息，而是帮助组块生长

### 实际含义
未来 schema 里应考虑：
- target chunk
- chunk dependencies
- prior knowledge assumptions

---

## 8. 概念地图是核心认知组件，不是附属图示

原因：
- 它让元素交互性变得可见
- 它是从“信息”到“结构化理解”的桥

### 实际含义
未来 concept map 不只是展示，还要成为：
- 回忆工具
- 追问工具
- 结构检查工具

---

## 9. 页面不是最终产品，课程结构才是最终产品

原因：
- 页面只是渲染形式
- 真正可复用的是课程组织逻辑与表达结构

### 实际含义
Engine 需要区分：
- engine layer
- course package layer
- module layer
- visual layer
- interaction layer
- validation layer

---

## 10. Agent 不是写手，是课程生产流水线的入口

原因：
- 一次性写作不稳定
- 教学质量需要多环节产出与 review

### 实际含义
Agent 的输出必须分阶段：
- topic framing
- curriculum graph
- lesson composition
- visual mapping
- QA critique
- human review

---

## 五、统一后的非目标（很重要）

下面这些事情，未来也许会做，但**绝不能成为当前设计思想的默认方向**。

### 非目标 1：知识大超市
不要变成什么都能塞的集合站。

### 非目标 2：大段 prose 的自动生成
不要把内容生产退化成“自动写文章”。

### 非目标 3：追求交互数量
交互不是越多越好，认知价值比数量重要。

### 非目标 4：假装支持无限主题
没有第二专题压力测试前，不要自嗨“通用引擎完成了”。

### 非目标 5：去掉人工审核
教学产品不是只做 factual QA。

---

## 六、对未来 `DESIGN.md v2` 的直接建议

未来 `DESIGN.md v2` 不该只是一份页面美术/组件规范，
而应升级成一份包含三层内容的文档：

### Layer A — Pedagogical principles
- 焦点问题
- 认知换挡
- 分解复杂 / 呈现复杂 / 穿透复杂
- worked example
- retrieval
- concept map role

### Layer B — Rendering principles
- narrative blocks
- visual blocks
- interaction capabilities
- cognitive load control
- block sequencing

### Layer C — Production principles
- schema-first
- validation-first
- engine/course separation
- workflow-first agent integration
- human review gate

---

## 七、一句话收束

把这些材料真正融进去之后，这个项目的设计思想已经不再是：

> “做一个 LLM 可视化学习网站”

而更接近：

> **建立一个复杂知识的认知升级系统。**

网站只是表现层，
课程是中间层，
设计思想和 schema 才是底层。
