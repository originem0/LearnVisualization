# DESIGN.md — Learning Site Engine 设计规范 v2

> 这不是页面美术说明，而是整个系统的设计约束。  
> 它定义：我们到底在做什么、不做什么、用什么原则生成课程、页面、图示、交互，以及未来 Agent 应该如何接入。

---

## 0. 产品定义

`LearnVisualization` 不是知识集合站，不是好看一点的文章站，也不是一键自动建站工具。

它要做的是：

> **把复杂知识转化成可被理解、可被穿透、可被迁移的学习体验。**

更准确地说：

> **它是一个复杂知识的可视化学习站引擎（Learning Site Engine）。**

它当前以 LLM 为主题样板，但目标不是停留在 LLM 专题站，而是抽象出：

- 课程结构
- 页面表达语法
- 概念可视化语法
- 交互能力系统
- 内容校验与发布流程
- Agent 生成入口

---

## 1. 非目标（必须反复确认）

下面这些方向，默认都不是我们的目标：

1. **不是知识大超市**  
   不做“什么都能塞”的集合站。

2. **不是内容工厂**  
   不追求大规模、低质量自动生成。

3. **不是纯文章站**  
   不把“写得不错的长文”误当成学习产品。

4. **不是交互游乐场**  
   交互不能为了酷炫存在，必须服务理解。

5. **不是全自动教学系统**  
   Agent 可以辅助生成，但不能取消人类教学判断。

6. **不是效率崇拜产品**  
   AI 的价值不是让我们更快地产出内容，而是放大问题感、结构感、判断力与创造性表达。

---

## 2. 顶层设计思想

这套系统由 5 条主线共同约束。

### 2.1 认知升级 / 换挡知识

有价值的知识不是信息增量，而是：

- 解释力强
- 能击穿旧直觉
- 能带来认知换挡

所以每章都必须帮助用户完成某种**框架改写**，而不是只是知道更多术语。

### 2.2 分解复杂、呈现复杂、穿透复杂

复杂主题难，不是因为字多，而是因为：

- 元素多
- 元素关系复杂
- 工作记忆无法同时处理

所以课程设计必须主动承担复杂性管理：

- 分解复杂
- 呈现结构
- 帮助穿透

### 2.3 概念分析、抽象与建模

好的学习不只是记住概念，而是：

- 澄清概念边界
- 看见概念所揭示的关系
- 从复杂现象中抽出关键要素
- 用模型组织这些关系

系统必须长期训练用户的不是“背定义”，而是：

- 概念分析
- 抽象
- 建模
- 缩放自如的心智地图

### 2.4 外脑 / 研究 / 知识工作流

Research 不是附件层，而是生产层的一部分。

系统未来必须支持：

- 搜索
- 抓取
- 提炼
- 联想
- resurfacing
- 转化成设计输入

也就是说，真正要管理的不是答案，而是问题、来源、结构与关联。

### 2.5 AI 时代的产品立场

AI 是基础能力，不是卖点本身。

随着外围技能越来越廉价，真正值钱的是：

- 问题感
- 结构判断
- 教学设计
- 视觉表达
- 创造性玩法

因此，AI 在这里的角色是：

> **结构化生产与认知放大器**

而不是“更快生成更多内容”的流水线机器。

## 3. 学习产品原则（Pedagogical Principles）

### 3.1 焦点问题原则

每个模块必须只有一个焦点问题。

要求：
- 焦点问题必须是学习驱动器，不是标题修辞
- 所有内容都围绕这个问题展开
- 焦点问题必须足够尖锐，能指向一个真正的认知冲突

### 3.2 反直觉开场原则

每章应优先击穿一个旧成见，而不是先给定义。

要求：
- 开场第一屏优先制造认知冲突
- 要告诉用户“你原来以为 X，其实不是”
- 定义可以滞后，但错觉必须先被打穿

### 3.3 结构先行原则

复杂内容必须先外显结构，再展开细节。

要求：
- 先给全貌，再给局部
- 先给关系，再给定义
- 先让用户知道“这一章里有哪些东西在互相作用”，再进入推导

### 3.4 Worked Example 原则

每章至少要有一个完整的 worked example / step-by-step 推导过程。

要求：
- 复杂机制必须拆成离散步骤
- 每一步只变化一个关键东西
- steps block 不是配菜，是理解主轴之一

### 3.5 前知识原则

影响学习的首要变量，是学习者已经知道什么。

要求：
- 模块应明确假定的前知识
- 如果前知识不足，要有桥接设计
- 不允许默认用户天然理解高抽象概念

### 3.6 组块化原则

课程的真实任务，是帮助用户形成更大的稳定认知组块。

要求：
- 每章应能回答“我在帮用户形成哪个组块？”
- 模块顺序应服务组块生长，而不是服务目录美观
- 每章都应该成为后续模块的支撑组块

### 3.7 主动回忆原则

看懂不等于学会。系统必须在合适位置触发 retrieval。

要求：
- 每 1–2 个模块应至少有一次轻量主动回忆
- retrieval 优先考关系、顺序、机制，而不是死记术语
- retrieval 形式可以是预测、补图、填空、重建步骤，而不只是 quiz

## 4. 内容表达原则（Content Expression Principles）

### 4.1 概念澄清优先

每章至少有一个需要被重新界定的关键概念。

要求：
- 说明它不是什么
- 说明它与近邻概念的区别是什么
- 说明它为什么重要：它揭示了什么关系

### 4.2 关系优先于定义

定义只有在揭示关系时才有价值。

要求：
- 不追求百科式定义
- 优先给出能帮助用户判断、比较、推理的定义
- 任何概念表达都要能回答：它让用户更看清了什么关系？

### 4.3 推导优先于宣告

不要把结论直接扔给用户，要让用户沿着推导路径走过去。

要求：
- narrative 应围绕“如何从问题出发推出答案”组织
- 正文是“为什么”的串联，不是“是什么”的堆砌

### 4.4 桥接原则

模块结尾必须自然导向下一个模块。

要求：
- bridge 不是“下节预告”文案，而是未解问题的自然延伸
- bridge 应基于当前章真正尚未解决的问题

---

## 5. 可视化原则（Visual Principles）

### 5.1 概念地图原则

概念地图不是插图，而是认知结构的外显工具。

要求：
- 每章应有概念关系图或同等功能的结构图
- 图优先服务焦点问题，不追求概念全收录
- 图的复杂度要受控，避免节点泛滥
- 图要帮助用户建立“缩小看全局、放大看局部”的心智地图

### 5.2 对比图原则

有差异的概念，不要只靠 prose 解释，必须并置比较。

要求：
- 左右 / A-B 对比清楚
- 保持其他条件不变，只突出关键差异
- 对比图要服务概念边界澄清

### 5.3 步骤图原则

复杂过程必须被做成视觉上的 step sequence。

要求：
- 每一步的变化清晰且单一
- 标签、强调点、视觉高亮都属于教学内容
- 默认由用户控制节奏

### 5.4 图文协同原则

图与文必须协同承担理解任务，而不是彼此重复。

要求：
- 图旁边必须就地解释
- 相关文字和图要近邻出现
- 能靠图解释清楚的，不重复讲一大段文字

---

## 6. 交互原则（Interaction Principles）

### 6.1 交互必须承担认知操作

交互存在的前提是：它服务于某种明确的认知动作。

允许的核心认知动作包括：
- compare
- step-through
- simulate
- trace
- classify
- retrieve
- manipulate parameter
- rebuild from map

### 6.2 控制节奏原则

复杂过程优先用户控制推进，而不是自动播放。

原因：
- 这是认知负荷理论中的 segmenting 在交互层的实现

### 6.3 即时反馈原则

用户动作之后必须立刻看到反馈。

要求：
- 不等待
- 不多层跳转
- 反馈本身应是教学信息

### 6.4 示例优先原则

用户输入前应有示例、默认值或预设按钮。

原因：
- 降低“我该输入什么”的启动成本
- 示例本身就是教学材料

---

## 7. Engine 抽象原则（System / Engine Principles）

### 7.1 页面不是本体

页面只是引擎的渲染结果，真正的本体是：
- course schema
- module schema
- visual schema
- interaction capability schema
- validation rules

### 7.2 Engine 与 Course Package 分离

必须区分：

#### Engine Layer
- layouts
- renderers
- interaction framework
- validation
- build / deploy

#### Course Package Layer
- course metadata
- audience
- learning goals
- module graph
- module content
- visual data
- interaction configuration

### 7.3 图优先于树

长期看，课程应被理解成依赖图，不只是线性目录树。

要求：
- 支持 prerequisite 关系
- 支持多路径学习
- 支持不同学习路径入口

### 7.4 Validation-first

任何生成、编辑、导入，最后都必须落入校验链。

校验的对象包括：
- 内容完整性
- narrative block 合法性
- schema integrity
- graph consistency
- visual / interaction reference existence
- prerender sanity

---

## 8. Agent 接入原则（Agent Integration Principles）

### 8.1 Workflow-first, Agent-assisted, Human-gated

整体流程必须先固定，再让 AI 在局部节点发挥。

### 8.2 Agent 不是写手，是课程生产入口

Agent 负责的不是“一次写成页面”，而是：
- topic framing
- audience / scope definition
- curriculum planning
- research synthesis
- lesson composition
- visual mapping
- critique / QA

### 8.3 结构化输出优先

Agent 默认输出应是结构化对象，而不是纯 prose。

### 8.4 Human Review Gate 不能取消

必须保留人工审核的环节，尤其是：
- 焦点问题是否尖锐
- 模块顺序是否合理
- 交互是否真的帮助理解
- 概念图是否过度或失焦
- 是否退化成“好看文章”

### 8.5 不加速垃圾内容

Agent 的使命不是提高内容产量，而是提高内容结构质量与设计质量。

---

## 9. 页面结构模板（当前模块页默认）

模块页默认顺序：

1. Hero（标签 / 模块编号 / 标题 / 副标题）
2. Quote（快速建立语境）
3. Focus Question（本章焦点问题）
4. Key Insight（本章关键洞察）
5. Chapter Route（本章路线）
6. Opening（反直觉开场）
7. Concept Map / Core Structure Visual
8. Narrative Body
   - comparison
   - steps
   - diagram
   - callout
   - worked examples
9. Hero/Secondary Interaction（视章节而定）
10. Bridge to Next Module
11. Deep Dive / structured metadata
12. Module Navigation

备注：
- 不是每章都必须有所有元素
- 但焦点问题、关键洞察、结构外显、worked example、bridge 是强约束

---

## 10. 一句话原则

> **先让用户看见结构，再让用户沿着步骤穿透复杂；AI 负责放大判断与创造，不负责替代教学质量。**
