## s02: Embedding 空间 — 从符号到语义向量

**焦点问题**: 一个整数怎么变成有语义的向量？

**反直觉开场**: Token ID 25105 和 25106 只差了 1，但它们代表的可能是"我"和一个生僻字。整数的大小关系毫无意义——模型需要另一种表示。

**交互组件 A — 向量算术试验场**:
- 用户选择/输入词，看对应的 2D 向量位置（降维可视化）
- 经典演示：king - man + woman ≈ queen
- 用户可以拖拽向量，直观感受"方向=语义关系"
- 预设示例：国王/王后/男人/女人、中国/北京/日本/东京
- 关键洞察：相似的词聚集在一起，类比关系是平行四边形

**交互组件 B — Embedding 查表动画**:
- 可视化 Embedding 矩阵：一个大表格，行=token ID，列=维度
- 用户输入一个 token ID，高亮对应行，抽取出向量
- 展示"查表"就是矩阵乘法的特例（one-hot × 矩阵 = 取一行）

**概念图**: Token ID → Embedding 矩阵（查表）→ 向量 → 语义空间（相似度/距离）→ 位置编码（绝对/旋转）→ 输入表示 → [→ s03 注意力]

**叙事推导**:
1. 为什么整数 ID 不够用（没有语义距离）
2. 查表思路：每个 ID 对应一个可学习的向量
3. 训练过程中向量自动组织（相似词靠近）
4. 位置编码：向量不知道自己在第几个位置
5. 正弦编码 vs 可学习编码 vs RoPE

**桥接**: 现在每个 token 有了语义向量，但它还是"自说自话"——它不知道上下文里其他词是什么。怎么让一个词"看见"其他词？

## s03: 注意力机制 — 一个词怎么知道该看哪个词

**焦点问题**: 一个词怎么决定该关注句子里的哪些其他词？

**反直觉开场**: "The cat sat on the mat because it was tired" —— "it" 指的是 cat 还是 mat？你一秒就知道答案。模型怎么做到的？不是靠语法规则，而是靠一个简单到离谱的操作：矩阵乘法。

**交互组件 A — 注意力热力图**:
- 用户输入一个句子（中/英），选择一个词
- 显示该词对所有其他词的注意力权重（热力图/条形图）
- 预设示例："The cat sat on the mat because it was tired"
- 点击不同的词，热力图实时变化
- 用颜色深浅表示权重大小

**交互组件 B — QKV 计算步进器**:
- 极简 3 词句子，展示 Q×K^T/√d → Softmax → ×V 的每一步
- 用户点"下一步"看矩阵运算过程
- 数字用颜色编码，大的红、小的蓝
- 展示因果遮罩的效果：toggle 一个开关，看到下三角遮罩

**概念图**: 输入向量 → Q/K/V 投影 → Q·K^T（相似度）→ √d_k 缩放 → 因果遮罩 → Softmax（概率）→ ×V（加权聚合）→ 输出 → 多头（并行多组 Q/K/V）→ [→ s04 Transformer]

**叙事推导**:
1. 从"看"的直觉出发——每个词需要决定看哪里
2. Query = "我在找什么"，Key = "我有什么"，Value = "我的内容"
3. Q·K 就是"匹配度"，为什么要除以 √d
4. Softmax 让权重变成概率分布
5. 因果遮罩：生成时只能看到前面的词
6. 多头注意力：不同的头关注不同类型的关系

**桥接**: 注意力让 token 之间有了交流。但光有注意力还不够——信息需要被处理、变换、一层层提炼。这就是 Transformer 的完整架构。

## s04: Transformer 结构 — 积木式的深度网络

**焦点问题**: 注意力层堆叠起来，为什么需要那么多"辅助零件"？

**反直觉开场**: Transformer 的核心论文标题是"Attention Is All You Need"——但如果你真的只用注意力层，模型会彻底崩溃。残差连接、层归一化、前馈网络——这些"配角"才是让整个系统稳定运行的关键。

**交互组件 A — 层级数据流模拟器**:
- 一个 Transformer block 的可视化：输入 → 注意力 → Add&Norm → FFN → Add&Norm → 输出
- 用户点击每个组件，看数据怎么流过
- 切换"残差连接开/关"，看输出如何变化（没有残差→梯度消失的直觉）
- 切换"层归一化开/关"，看数值如何爆炸/稳定

**交互组件 B — 架构对比器**:
- Encoder-Decoder vs Decoder-only 并排对比
- 用户切换查看每种架构的数据流
- 标注哪些模型用哪种架构（BERT=Encoder, GPT=Decoder-only, T5=Enc-Dec）

**概念图**: 输入 → [Attention + Residual + LayerNorm] → [FFN + Residual + LayerNorm] → 输出 = 1 个 Block → N 层堆叠 → Encoder/Decoder/Decoder-only → [→ s05 预训练]

## s05: 预训练目标 — 自监督学习的巧妙设计

**焦点问题**: 没有人给数据打标签，模型怎么从海量文本中自己学会语言？

**反直觉开场**: GPT 的训练目标简单得离谱：给定前面的词，预测下一个词。就这么一个任务，用几 TB 的文本反复训练，居然学出了推理、翻译、写代码的能力。这怎么可能？

**交互组件 A — 下一个词预测游戏**:
- 显示一段文字，最后一个词被遮住
- 用户猜下一个词（给 4 个选项）
- 揭晓后展示概率分布（不只是对错，而是模型认为每个词的可能性）
- 多轮游戏，从简单（"天空是___色的"）到困难（"经济学家认为通货膨胀的根本原因是___"）
- 关键洞察：简单句模型很确定（概率集中），复杂句模型更分散

**交互组件 B — MLM vs CLM 对比器**:
- 同一个句子，展示两种预训练方式的差异
- CLM（因果）：从左到右逐词预测，遮住右边所有词
- MLM（掩码）：随机遮住 15% 的词，双向看
- 用户 toggle 切换，直观看到信息流的差异

**概念图**: 原始文本 → 自监督信号（不需要标签）→ CLM（预测下一个词）/ MLM（预测被遮的词）→ 交叉熵损失 → 反向传播 → 参数更新 → [→ s06 损失与优化]

## s06: 损失与优化 — 梯度如何塑造语言能力

**焦点问题**: 一个"猜错了"的信号，怎么变成让模型变聪明的力量？

**反直觉开场**: 模型做了一次预测，错了。这个"错"被量化成一个数字（损失），然后这个数字倒着流回整个网络，告诉每一个参数"你应该往哪个方向调一点点"。几十亿个参数，每个调一点点——语言能力就这样涌现了。

**交互组件 A — 损失地形探索器**:
- 2D 等高线图表示损失曲面
- 一个小球代表当前参数位置
- 用户可以：调整学习率（步长大小）、点击"走一步"看梯度下降
- 学习率太大→跳过最优点；太小→卡在局部最小值
- 预设场景：平滑谷地、陡峭悬崖、鞍点

**交互组件 B — 学习率调度可视化**:
- X 轴是训练步数，Y 轴是学习率
- 用户切换不同调度策略（constant、cosine、warmup+decay）
- 实时看到学习率曲线变化

**概念图**: 预测 → 与真实比较 → 交叉熵损失 → 反向传播（链式法则）→ 梯度 → 优化器（SGD/Adam）→ 学习率调度 → 参数更新 → [→ s07 微调]

---

## s07: 微调与对齐 — 从通用到可控

**焦点问题**: 预训练出来的模型只会"接话"——怎么让它变成一个有用的助手？

**反直觉开场**: GPT-3 预训练完后，你问它"法国的首都是什么？"，它可能回答"德国的首都是什么？意大利的首都是什么？"——因为它的训练目标是续写，不是回答问题。从"续写机器"到"智能助手"，还有一段关键的路要走。

**交互组件 A — Base vs SFT vs RLHF 输出对比**:
- 同一个 prompt，展示三种模型的典型输出风格
- Base model：续写风格，可能跑题
- SFT：格式正确但可能机械
- RLHF：自然、有帮助、拒绝有害请求
- 用户切换不同 prompt 和不同模型阶段

**交互组件 B — LoRA 参数计算器**:
- 用户设置：原始维度、秩 r、层数
- 实时计算全量微调 vs LoRA 的参数量对比
- 可视化：一个大矩阵被分解成两个小矩阵的乘积

**概念图**: 预训练模型 → SFT（指令微调）→ RLHF（人类反馈强化学习）→ 奖励模型 + PPO → 对齐的模型 | 参数高效方法：LoRA / QLoRA → [→ s08 Prompt]

## s08: Prompt Engineering — 用自然语言编程

**焦点问题**: 同一个模型，为什么换个问法就能得到完全不同质量的回答？

**反直觉开场**: 你说"写一首诗"，模型给你一首平庸的诗。你说"你是李白，酒后望月，写一首七言绝句"，模型给你一首惊艳的诗。模型没变，变的只是你的 prompt。Prompt 就是给模型的"前知识"。

**交互组件 A — Prompt 工坊**:
- 左边：prompt 模板（角色 + 上下文 + 任务 + 格式 + 示例）
- 用户可以 toggle 开/关每个部分
- 右边：模拟的输出质量评分/长度/风格变化（预设，非实时调用 API）
- 展示：加上 "step by step" 后输出如何变化
- 展示：few-shot 示例数量对输出的影响

**交互组件 B — Few-shot 示例构建器**:
- 一个任务（如：情感分类）
- 用户拖拽添加示例（0-shot, 1-shot, 3-shot）
- 看到"模型理解度"指标的变化（模拟）

**概念图**: 用户意图 → Prompt 构成（角色/上下文/任务/格式/示例）→ 技巧（CoT/Few-shot/Self-consistency）→ 模型输出 → 输出质量 → [→ s09 Scaling]

## s09: Scaling Laws — 规模的力量与极限

**焦点问题**: 更大的模型一定更好吗？更多的数据一定更好吗？

**反直觉开场**: OpenAI 发现了一个惊人的规律：模型的损失和参数量、数据量、算力之间存在精确的幂律关系。这意味着——在训练之前，你就能预测模型会有多好。

**交互组件 A — Scaling Law 绘图器**:
- 双对数坐标，X 轴=参数量/数据量/算力，Y 轴=损失
- 用户拖动滑块改变参数量、数据量
- 看到损失曲线的变化
- 标注真实模型的位置（GPT-2/3/4, Llama 等）
- Chinchilla 最优线：给定算力预算，最优的参数/数据比

**交互组件 B — 训练预算计算器**:
- 输入：GPU 数量、GPU 型号、训练天数
- 输出：总 FLOPs、推荐参数量、推荐数据量（按 Chinchilla）
- 对比真实模型的训练配置

**概念图**: 参数量(N) + 数据量(D) + 算力(C) → 幂律关系 → 损失预测 → Chinchilla 最优比 → 训练规划 → [→ s10 涌现]

## s10: 涌现与推理 — 量变何时引发质变

**焦点问题**: 模型变大，能力是渐进提升还是突然出现？

**反直觉开场**: 一个 10 亿参数的模型完全无法做三位数加法。一个 100 亿的还是不行。但到 1000 亿——突然就会了。准确率不是从 50% 涨到 60% 涨到 70%，而是从接近 0% 直接跳到 90%+。这就是"涌现"。

**交互组件 A — 涌现能力台阶图**:
- X 轴=模型大小（对数），Y 轴=任务准确率
- 多条曲线代表不同任务（加法、翻译、推理、代码）
- 用户拖动"模型大小"滑块，看哪些能力在哪个点"解锁"
- 有些任务是平滑增长（信息检索），有些是阶梯式跳变（推理）

**交互组件 B — Chain-of-Thought 开关**:
- 一个推理题（如："小明有 3 个苹果..."）
- Toggle：直接回答 vs 逐步推理
- 展示两种模式下的答案差异和正确率

**概念图**: 模型规模增长 → 平滑提升（记忆/流畅度）+ 突然涌现（推理/少样本学习）→ 涌现能力 → Chain-of-Thought → 推理能力 → [→ s11 上下文]

## s11: 上下文窗口 — 长度与记忆的权衡

**焦点问题**: 为什么模型不能一次读完一本书？

**反直觉开场**: GPT-4 的上下文窗口是 128K token——大约 10 万字，一本小说的长度。听起来很大？但注意力机制的计算量和长度的平方成正比。128K 的注意力矩阵有 160 亿个元素。每一层、每一个头都要算一遍。

**交互组件 A — 上下文长度计算器**:
- 用户输入文本长度（或拖动滑块）
- 实时显示：注意力矩阵大小（n²）、显存占用、计算时间估算
- 标注不同模型的窗口大小（GPT-4: 128K, Claude: 200K, Gemini: 1M）
- 直观可视化：用方格表示注意力矩阵，n 增大时方格面积爆炸性增长

**交互组件 B — "什么能装进上下文" 计算器**:
- 输入不同类型的内容（一封邮件≈500 tokens, 一篇论文≈8K, 一本书≈100K）
- 看它们能不能装进不同模型的上下文窗口
- 积木式可视化：不同大小的方块填充一个容器

**概念图**: 上下文窗口 → 注意力矩阵 O(n²) → 显存/计算瓶颈 → 解决方案（稀疏注意力/滑动窗口/RoPE 外推）→ 长文本能力 → [→ s12 回顾]

## s12: 系统化回顾 — 将模型变成自己的知识

**焦点问题**: 从文本到智能，完整的路径是什么？

**反直觉开场**: 你现在知道了 LLM 的每一个零件。但零件和机器是两回事。让我们把所有东西串起来——追踪一个 prompt 从输入到输出的完整旅程。

**交互组件 A — 全流水线追踪器**:
- 输入一个 prompt
- 可视化它经过所有阶段：Tokenize → Embed → Attention×N → FFN×N → Output → Sample
- 每个阶段可以点击展开细节
- 数据在每个阶段的形状变化：字符串 → [token IDs] → [vectors] → [transformed vectors] → [probabilities] → 输出文字
- 链接回每个模块的详情页

**交互组件 B — 知识网络图**:
- 12 个模块作为节点，展示它们之间的概念连接
- 用户点击任意连接线，看到两个模块之间的关键关系
- 可作为整个课程的导航地图

**概念图**: 输入文本 → s01 分词 → s02 嵌入 → s03 注意力 → s04 Transformer → [s05 预训练 → s06 优化 → s07 对齐] → s08 Prompt → 输出 | 全局视角：s09 规模 → s10 涌现 → s11 上下文
